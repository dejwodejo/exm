Bezpieczeństwo IT

Bezpieczeństwo systemów i aplikacji

Notatka egzaminacyjna

Bezpieczeństwo IT to ochrona aktywów informacyjnych przed zagrożeniami przy zachowaniu funkcjonalności systemu. Projektowanie zaczyna się od modelu zagrożenia (threat model): co chronimy, przed kim, jakie są ścieżki ataku. Bez tego mechanizmy obronne są dobierane losowo.

Triada CIA i model dostępu

Triada CIA (Confidentiality, Integrity, Availability):

  • Poufność (Confidentiality) — dane widoczne tylko dla uprawnionych (szyfrowanie, kontrola dostępu).
  • Integralność (Integrity) — wykrywanie nieautoryzowanych modyfikacji (skróty, podpisy, MAC — Message Authentication Code).
  • Dostępność (Availability) — system odpowiada na żądania (redundancja, anty-DDoS, backup).

Często rozszerzane o autentyczność (potwierdzenie tożsamości) i niezaprzeczalność (nadawca nie może zaprzeczyć podpisu — Autoryzacja i podpis).

Macierz kontroli dostępu Lampsona M:S×O→2RM: S \times O \to 2^{R}M:S×O→2R — formalny model: dla podmiotu s∈Ss\in Ss∈S (subject) i obiektu o∈Oo\in Oo∈O (object) komórka M(s,o)⊆RM(s,o)\subseteq RM(s,o)⊆R przechowuje podzbiór zbioru wszystkich praw R={read,write,execute,… }R=\{\text{read},\text{write},\text{execute},\dots\}R={read,write,execute,…} (2R2^R2R = zbiór potęgowy, czyli wszystkie podzbiory RRR). Implementacje: ACL (Access Control List — kolumna macierzy, przechowywana przy obiekcie) vs capabilities (wiersz macierzy, przy podmiocie).

Paradoks urodzin obniża koszt kolizji do ≈2n/2\approx 2^{n/2}≈2n/2 (efektywny poziom bezpieczeństwa =n/2=n/2=n/2 bitów), więc SHA-1 (Secure Hash Algorithm, n=160n=160n=160, kolizja ≈280\approx 2^{80}≈280) jest złamany, a SHA-256 (n=256n=256n=256, ≈2128\approx 2^{128}≈2128) zostaje bezpieczny.

Szyfr symetryczny — trójka (Gen,Enc,Dec)(\text{Gen}, \text{Enc}, \text{Dec})(Gen,Enc,Dec) (generacja klucza / szyfrowanie / deszyfrowanie) z jednym kluczem; szybki, AES (Advanced Encryption Standard) — standard. Szyfr asymetryczny — para (pk,sk)(pk, sk)(pk,sk) klucz publiczny/prywatny, oparty o problem trudny (faktoryzacja w RSA — Rivest-Shamir-Adleman; logarytm dyskretny w ECC — Elliptic Curve Cryptography); wolny, używany do wymiany kluczy i podpisów.

MAC (Message Authentication Code) — funkcja MACk(m)\text{MAC}_k(m)MACk​(m) zapewnia integralność + autentyczność z kluczem symetrycznym. HMAC (Hash-based MAC) = standardowa konstrukcja MAC z funkcji skrótu. Zwykły skrót integralności bez klucza nie autentykuje (atakujący może zmienić wiadomość i przeliczyć hash).

KDF (Key Derivation Function) — przekształca hasło w klucz: f(password,salt,iter)→kf(\text{password}, \text{salt}, \text{iter}) \to kf(password,salt,iter)→k. PBKDF2 (Password-Based KDF 2), bcrypt, scrypt, Argon2id (zwycięzca Password Hashing Competition, memory-hard). Sól (per-user, przechowywana z hashem) blokuje rainbow tables; pieprz (globalny sekret poza bazą) chroni gdy wycieknie tylko baza. Hashowanie haseł SHA-256 z solą jest zbyt szybkie — GPU robi miliardy hashów/s.

Częsta pomyłka: szyfrowanie (odwracalne z kluczem) ≠ hashowanie (jednokierunkowe) ≠ kodowanie (Base64 — odwracalne bez klucza, nie chroni nic).

Szyfrowanie at rest / in transit / in use

  • At rest — dane na dysku: LUKS (Linux Unified Key Setup), BitLocker, szyfrowanie kolumn w bazie (TDE — Transparent Data Encryption).
  • In transit — w sieci: TLS 1.3 (Transport Layer Security), IPsec (Internet Protocol Security), VPN (Virtual Private Network).
  • In use — w pamięci: trusted execution environment (Intel SGX — Software Guard Extensions, AMD SEV — Secure Encrypted Virtualization), MPC (Multi-Party Computation), FHE (Fully Homomorphic Encryption). SGX podważone przez ataki side-channel (Foreshadow).

Bezpieczeństwo systemu operacyjnego

Pierścienie ochrony (rings): formalna architektura x86 definiuje tylko Ring 0 (kernel mode — pełny dostęp do sprzętu) oraz Ring 3 (user mode — wymaga syscalla). Nieformalnie mówi się też o "ujemnych pierścieniach" (rozszerzenie popularnonaukowe, nie część specyfikacji sprzętowej): Ring -1 (hypervisor), Ring -2 (SMM — System Management Mode), Ring -3 (Intel ME — Management Engine). Atak na niższy pierścień kompromituje wszystkie wyższe.

Kontrola dostępu w OS: DAC (Discretionary Access Control) — uprawnienia POSIX rwx, właściciel decyduje, podatny na trojany (proces dziedziczy uprawnienia użytkownika). MAC (Mandatory Access Control — uwaga: inny skrót niż kryptograficzny MAC = Message Authentication Code) — SELinux, AppArmor; polityka systemowa nadpisuje wolę właściciela. (Modele DAC/MAC/RBAC/ABAC szczegółowo w Autoryzacji i podpisie.)

Izolacja procesów — rosnące poziomy:

  • Sandbox (seccomp-bpf — filtrowanie syscalli, AppArmor) — w tym samym kernelu.
  • Kontener (Docker, podman; namespaces + cgroups) — dzieli kernel z hostem; nie jest granicą bezpieczeństwa multi-tenant.
  • Maszyna wirtualna (VM, KVM — Kernel-based Virtual Machine / Xen) — własny kernel, izolacja na poziomie hypervisora; najsilniejsza, ale podatna na VM escape (Venom).

Hardening pamięci:

  • ASLR (Address Space Layout Randomization) — randomizuje adresy, blokuje stałe offsety w eksploitach.
  • DEP/NX (Data Execution Prevention / No-eXecute), zasada W ⊕ X (write XOR execute) — strona albo zapisywalna, albo wykonywalna; blokuje shellcode w buforze.
  • Stack canaries — wartość kontrolna przed adresem powrotu, wykrywa stack overflow.
  • CFI (Control Flow Integrity) — weryfikuje legalność celów wywołań pośrednich; razem z ASLR blokuje ROP (Return-Oriented Programming — łańcuchy istniejących gadgetów omijających DEP).

Zasada najmniejszych przywilejów (Saltzer-Schroeder) — proces ma tylko uprawnienia niezbędne do zadania. SUID (Set User ID) — bit pozwalający binarce działać z uprawnieniami właściciela pliku; wektor lokalnej eskalacji (GTFOBins). Linux capabilities (CAP_NET_BIND_SERVICE itd.) drobnoziarniście zastępują roota.

TCB (Trusted Computing Base) — minimalny zbiór komponentów (kernel, firmware, sterowniki) których kompromitacja łamie politykę bezpieczeństwa. Secure Boot weryfikuje podpisy bootloadera; TPM (Trusted Platform Module) mierzy hashe komponentów do PCR (Platform Configuration Register) i służy jako root of trust dla attestation.

Aktualizacje i łatanie — wektor odpowiedzialny za większość udanych ataków masowych (WannaCry przez nieparowaną SMB — Server Message Block).

Bezpieczeństwo aplikacji — OWASP Top 10 (2025; 2021 wciąż często w sylabusach)

OWASP (Open Worldwide Application Security Project) — niekomercyjna organizacja publikująca standardy bezpieczeństwa aplikacji webowych. Top 10 2025 jest aktualnym wydaniem; lista 2021 nadal bywa spotykana w sylabusach, więc traktuj ją jako historyczną bazę i porównuj z aktualizacją 2025:

  1. A01 Broken Access Control — IDOR (Insecure Direct Object Reference), brak weryfikacji uprawnień.
  2. A02 Cryptographic Failures — słaba kryptografia, brak szyfrowania.
  3. A03 Injection — SQLi (SQL Injection), command injection, LDAP (Lightweight Directory Access Protocol) injection.
  4. A04 Insecure Design — brak modelu zagrożenia w fazie projektu.
  5. A05 Security Misconfiguration — domyślne hasła, otwarte porty.
  6. A06 Vulnerable & Outdated Components — biblioteki z CVE (Common Vulnerabilities and Exposures).
  7. A07 Authentication Failures — słabe hasła, brak rate limiting.
  8. A08 Software & Data Integrity Failures — supply chain.
  9. A09 Logging & Monitoring Failures — niewykryte włamania.
  10. A10 SSRF (Server-Side Request Forgery) — serwer pobiera URL podany przez atakującego.

Klasyczne ataki aplikacyjne

  • SQL Injection — wstrzyknięcie SQL (Structured Query Language) przez nieoczyszczone wejście (' OR 1=1 --). Obrona: prepared statements / parametryzowane zapytania (nie konkatenacja stringów), ORM, najmniejsze uprawnienia konta DB.
  • XSS (Cross-Site Scripting) — wariant stored / reflected / DOM-based (Document Object Model); wstrzyknięcie JS (JavaScript) w stronę ofiary. Obrona: escape kontekstowy (HTML/JS/URL), CSP (Content Security Policy), flagi HttpOnly/Secure na ciasteczkach. SQLi atakuje serwer, XSS przeglądarkę ofiary.
  • CSRF (Cross-Site Request Forgery) — wykorzystanie aktywnej sesji ofiary do akcji bez jej wiedzy. Obrona: tokeny anty-CSRF, ciasteczka SameSite, weryfikacja Origin/Referer.
  • Buffer overflow — stack overflow nadpisuje return address (kontrola przepływu od razu); heap overflow nadpisuje metadane alokatora (eksploatacja pośrednia). Obrona: stack canaries, DEP, ASLR, CFI, języki memory-safe (Rust, Go).
  • TOCTOU (Time-Of-Check Time-Of-Use) — race condition między sprawdzeniem warunku a użyciem zasobu (access() → open() z podmianą symlinka). Obrona: atomowe operacje (O_CREAT|O_EXCL, openat).
  • Supply chain — typosquatting w npm/PyPI, kompromitacja CI/CD (Continuous Integration / Continuous Deployment) (SolarWinds 2020, xz-utils backdoor 2024). Obrona: SBOM (Software Bill of Materials), lockfiles, podpisywanie artefaktów (Sigstore), skanery zależności.

Malware, phishing, DDoS

Taksonomia malware:

  • Wirus — wymaga nosiciela (infekuje pliki).
  • Robak — propaguje się autonomicznie przez sieć (Morris, WannaCry).
  • Trojan — udaje legalny program.
  • Rootkit — ukrywa obecność modyfikując kernel/syscalle (kernel rootkit — Ring 0, niewidoczny dla AV — Antivirus w userspace).
  • Ransomware — szyfruje dane i żąda okupu (CryptoLocker; NotPetya był w istocie wiperem podszywającym się pod ransomware — szyfrował bezpowrotnie, okup to fasada).
  • RAT (Remote Access Trojan), keylogger, cryptojacker, fileless malware (living-off-the-land przez PowerShell/WMI — Windows Management Instrumentation).

Phishing — spear phishing (celowany), BEC (Business Email Compromise), smishing (SMS — Short Message Service), vishing (głos). Obrona: szkolenia, MFA (Multi-Factor Authentication) odporne na phishing (FIDO2 — Fast IDentity Online 2 / WebAuthn), filtry mailowe (DMARC — Domain-based Message Authentication, Reporting & Conformance; SPF — Sender Policy Framework; DKIM — DomainKeys Identified Mail).

DDoS (Distributed Denial of Service) — typy: volumetric (zalanie pasma), protocol (SYN flood — wykorzystuje 3-way handshake TCP), application-layer (HTTP flood). Obrona: CDN (Content Delivery Network), anty-DDoS, rate limiting, scrubbing center.

Backup i disaster recovery

  • Strategia 3-2-1 — 3 kopie danych, 2 różne nośniki, 1 kopia offsite.
  • RTO (Recovery Time Objective) — maksymalny dopuszczalny czas niedostępności po incydencie.
  • RPO (Recovery Point Objective) — maksymalna dopuszczalna utrata danych mierzona w czasie (np. RPO=1h ⇒ backup co godzinę).
  • Testowanie restore'ów (backup nieprzetestowany = brak backupu).
  • Przeciwko ransomware: kopie immutable / air-gapped / WORM (Write Once Read Many).

Narzędzia obronne i standardy

  • Firewall sieciowy (iptables/nftables, pf) i WAF (Web Application Firewall) — ModSecurity dla aplikacji webowych.
  • IDS/IPS (Intrusion Detection/Prevention System) — Snort, Suricata.
  • SIEM (Security Information and Event Management) — Splunk, ELK (Elasticsearch + Logstash + Kibana); agregacja logów i korelacja zdarzeń.
  • EDR (Endpoint Detection and Response) — monitoring stacji końcowych.
  • IC11 — blokowanie IP po powtórzonych próbach logowania.
  • CVE — globalny katalog podatności; CVSS (Common Vulnerability Scoring System) — punktacja krytyczności 0–10.
  • MITRE ATT&CK (Adversarial Tactics, Techniques & Common Knowledge) — taksonomia taktyk i technik atakujących.

Powiązania z innymi zagadnieniami

  • Autoryzacja, podpis — bezpośrednia kontynuacja: modele DAC/MAC/RBAC/ABAC, MFA, PKI (Public Key Infrastructure), podpis cyfrowy, eIDAS.
  • Preprocessing — anonimizacja, pseudonimizacja, maskowanie danych wrażliwych (RODO / GDPR) realizują wymóg poufności.
  • ML / sieci neuronowe — ataki adversarialne, model poisoning, model stealing, membership inference; specyfika bezpieczeństwa ML.
  • Porównanie narzędzi — wybór narzędzi determinowany przez wymagania bezpieczeństwa i certyfikacje (ISO 27001, SOC 2 — Service Organization Control 2).
