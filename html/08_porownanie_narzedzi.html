Big Data i systemy rozproszone

Porównanie metod i narzędzi

Notatka egzaminacyjna

1. Definicje fundamentalne

  • Przetwarzanie wsadowe (batch) — funkcja \(f: D^* \to Y\), gdzie \(D\) to dziedzina pojedynczego rekordu, \(D^*\) (gwiazdka Kleene'ego) to zbiór wszystkich skończonych kolekcji rekordów, a \(Y\) to przestrzeń wyników; całe wejście jest znane na starcie (rozmiar dowolny, ale skończony).
  • Przetwarzanie strumieniowe (stream) — funkcja \(f: (D\times T)^\omega \to Y^\omega\) na nieskończonej (\(\omega\) = ciąg nieskończony) sekwencji par (zdarzenie \(d\in D\), znacznik czasu \(t\in T\)). Kluczowa własność: \(f\) jest przyczynowa — \(i\)-ty element wyjścia zależy tylko od prefiksu wejścia \((d_1,t_1),\dots,(d_i,t_i)\), nie od przyszłych zdarzeń; wynik powstaje przyrostowo (on-line), a nie z całego ciągu naraz.
  • Czas zdarzenia (event time) vs czas przetwarzania (processing time) — pierwszy to moment, w którym zdarzenie zaszło; drugi to moment, w którym dotarło do systemu.
  • Watermark \(W(t)\) — niemalejąca funkcja czasu przetwarzania; \(W(t)\) to systemowa estymata lub deklarowana dolna granica event-time: system zakłada, że wszystkie zdarzenia o event-time \(\leq W(t)\) już dotarły. Zdarzenie o event-time \(e\) jest spóźnione (late), gdy napływa już po tym, jak \(W(t)>e\).
  • Okna: tumbling (rozłączne, równe), sliding (zachodzące), session (oddzielone okresami nieaktywności).
  • OLTP (Online Transaction Processing) — wiele krótkich transakcji wierszowych (INSERT/UPDATE pojedynczych rekordów). OLAP (Online Analytical Processing) — duże agregaty po milionach wierszy.
  • ACID (Atomicity, Consistency, Isolation, Durability) — gwarancje transakcji. BASE (Basically Available, Soft state, Eventual consistency) — luźniejsze, typowe dla NoSQL.
  • Twierdzenie CAP (Brewer): w systemie rozproszonym nie da się zagwarantować jednocześnie wszystkich z Consistency, Availability, Partition tolerance. PACELC (Abadi; if Partition then Availability vs Consistency, Else Latency vs Consistency) dodaje wymiar normalnej pracy: latencja vs spójność.
  • Gwarancje dostarczenia: at-most-once (możliwa utrata), at-least-once (możliwe duplikaty), exactly-once (każde zdarzenie przetworzone dokładnie raz — najtrudniejsze, w praktyce często "effectively-once").

2. Osie porównania

Wymiary, wzdłuż których porównujemy narzędzia:

  • Skala danych: GB (single-node) → TB (single-node z formatem kolumnowym) → PB (rozproszony klaster).
  • Latencja: minuty–godziny (batch) → sekundy (micro-batch) → milisekundy (true streaming) → < 10 ms (OLTP).
  • Model danych: relacyjny (wiersz), kolumnowy, dokumentowy, klucz-wartość, grafowy.
  • Schemat: schema-on-write (hurtownia, RDBMS — Relational Database Management System) vs schema-on-read (data lake).
  • Tryb transformacji: ETL (Extract–Transform–Load, transformacja przed zapisem) vs ELT (transformacja w docelowej hurtowni; dominuje od czasu tanich hurtowni chmurowych).
  • Architektura zasobów: shared-memory (jeden węzeł), shared-disk (klaster + wspólny storage), shared-nothing (klaster z lokalnymi dyskami — Hadoop, Spark).

Czysty MR jest dziś legacy; paradygmat shuffle&sort został.

4. Formaty danych i gwarancje

  • Parquet / ORC — kolumnowe, kompresja, predicate pushdown, late materialization. Standard w analityce (BigQuery, Snowflake, Spark, lakehouse).
  • Apache Avro — wierszowy, ze schemą; standard w Kafka + Schema Registry.
  • CSV / JSON — czytelne, ale nieefektywne (brak schematu, brak kompresji kolumnowej, wolna deserializacja).
  • Vectorized execution (ClickHouse, DuckDB, Photon w Databricks) wykorzystuje SIMD (Single Instruction Multiple Data) i cache-friendliness — stąd przewaga kolumnowych silników w OLAP.
  • Exactly-once w streamingu wymaga albo idempotentnych sinków, albo transakcyjnych commitów (Kafka transactions, distributed snapshots Chandy–Lamporta we Flink, two-phase commit). Sam fakt, że narzędzie reklamuje "exactly-once", nie wystarczy — sink i źródło muszą to wspierać.

5. Architektury referencyjne

  • Lambda architecture: równoległa warstwa batch (Spark) + warstwa speed (Flink/Spark Streaming); duplikacja logiki, ale spójność wyników historycznych.
  • Kappa architecture: tylko streaming z replayowalnym logiem (Kafka); batch = szczególny przypadek długiego replayu. Prostsza operacyjnie w systemach event-streaming, ale nie dominuje uniwersalnie; Lambda nadal bywa sensowna przy ciężkich wsadach i niezależnej warstwie serving.

Separacja compute/storage (object storage S3/GCS zamiast HDFS) podważyła znaczenie data locality — silniki jak Presto/Trino, Snowflake, BigQuery liczą na szybką sieć, nie na lokalny dysk.

6. Heurystyka decyzyjna

Wybór narzędzia jest problemem decyzyjnym wielokryterialnym — kryteria: skala, latencja, budżet, kompetencje zespołu, dojrzałość ekosystemu, ryzyko vendor lock-in:

  • < 10 GB, batch, jeden analityk → pandas/DuckDB.
  • 10 GB – 1 TB, batch → Polars/DuckDB lokalnie albo hurtownia chmurowa + dbt.

  • 1 TB batch → Spark + lakehouse (Delta/Iceberg) albo BigQuery/Snowflake.

  • streaming z latencją < 1 s → Flink lub Kafka Streams; > 1 s wystarczy Spark Structured Streaming.
  • zespół SQL-only → hurtownia + dbt; zespół Python → Spark/Polars/Dask.
  • OLTP z silną spójnością → PostgreSQL; rozproszone OLTP → NewSQL (Spanner, CockroachDB).

Modele kosztowe są częścią wyboru: BigQuery (per-query bytes scanned), Snowflake (per-second compute), self-hosted Spark (TCO — Total Cost of Ownership infrastruktury). Optymalizacja kosztu = partycjonowanie, klastrowanie, materialized views, dobór rozmiaru warehouse.

7. Powiązania z innymi zagadnieniami

  • Big Data: dobór narzędzia do 5V.
  • MapReduce: historyczny punkt odniesienia każdego silnika.
  • Preprocessing: skala decyduje (pandas vs Spark vs SQL).
  • Modele wielowymiarowe/Porównanie modelowania: OLAP, gwiazda/płatek → hurtownia kolumnowa.
  • Raportowanie: BI (Business Intelligence — Tableau, Power BI, Looker, Superset) nad hurtownią/lakehouse.
  • Text mining: Spark NLP/Elasticsearch vs spaCy/NLTK lokalnie.
  • Klasyczne ML/Sieci neuronowe: scikit-learn vs Spark MLlib vs TensorFlow/PyTorch (GPU).
  • Bezpieczeństwo IT: Kerberos, RBAC (Role-Based Access Control), szyfrowanie at-rest/in-transit.
  • Optymalizacja wielokryterialna: MCDM (Multi-Criteria Decision Making) — AHP (Analytic Hierarchy Process), TOPSIS (Technique for Order of Preference by Similarity to Ideal Solution), Pareto.
