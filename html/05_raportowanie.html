Techniki raportowania

Notatka egzaminacyjna

Raport to ustrukturyzowana prezentacja danych przekształconych w informację dla odbiorcy decyzyjnego. Cztery osie projektowe: forma, świeżość, sterowanie, odbiorca (rozwinięte w tabeli niżej) — wyznaczają wybór technologii.

Definicje

Miara — wielkość liczbowa raportowana; addytywna (sumuje się po wszystkich wymiarach, np. sprzedaż) lub póładdytywna (sumuje się po niektórych wymiarach, ale nie po czasie, np. stan magazynu / saldo konta). Wymiar — atrybut kategoryczny do agregacji (region, czas). KPI (Key Performance Indicator) — skalarna funkcja miar \(\text{KPI} = g(M_1, \ldots, M_k)\), gdzie \(M_1,\ldots,M_k\) to miary, a \(g:\mathbb{R}^k\to\mathbb{R}\) (np. konwersja \(=\) transakcje/sesje, czyli \(g(a,b)=a/b\)).

Funkcja agregująca \(f:\mathcal{M}(D)\to\mathbb{R}\) odwzorowuje multizbiór wartości kolumny w liczbę; \(D\) to dziedzina wartości agregowanej miary, \(\mathcal{M}(D)\) — zbiór skończonych multizbiorów nad \(D\). Duplikaty są istotne: \(\text{SUM}(1,1,1)=3\neq\text{SUM}(1)=1\) (dziedziną jest multizbiór, nie zbiór).

Klasy Graya (taksonomia kostki danych):

  • distributive (SUM, MIN, COUNT) — wynik globalny liczy się z wyników cząstkowych ⇒ można pre-agregować i cachować;
  • algebraic (AVG) — potrzebna stała liczba agregatów pomocniczych (para sum, count);
  • holistic (MEDIAN, COUNT DISTINCT) — brak dokładnej pre-agregacji w stałej pamięci (istnieją aproksymacje, np. HyperLogLog).

Stąd sumę można cachować, mediany nie; średnia globalna to \(\dfrac{\sum_i \text{sum}_i}{\sum_i \text{count}_i}\neq \dfrac{1}{g}\sum_{i=1}^{g}\text{mean}_i\) (gdzie \(g\) = liczba grup) — średnia ze średnich zawodzi przy nierównych licznościach grup i dlatego AVG jest agregatem algebraicznym.

Funkcja okienkowa — agregacja na partycji wierszy (PARTITION BY) z ramką (ROWS BETWEEN); umożliwia rankingi, sumy kroczące, porównania rok-do-roku (LAG) bez kolapsu wierszy.

Determinizm (własność obliczenia) — ten sam wsad + ten sam kod + to samo środowisko ⇒ ten sam wynik. Wymaga wersjonowania czterech elementów: kodu, danych, środowiska (zależności), losowości (seed). Repeatability (własność procesu) — ten sam zespół powtarza ten sam eksperyment na tym samym stanowisku.

Reproducibility w sensie naukowym (definicja ACM) — odtworzenie wyniku przez inny zespół na innym systemie/środowisku — najbardziej restrykcyjne z tych pojęć.

Typologia raportów

Oś Warianty
Forma pixel-perfect (faktury) · dashboard interaktywny · raport literacki (kod+narracja) · ad-hoc query
Świeżość statyczny · batch (nightly) · live (streaming)
Sterowanie predefiniowany (centralny) · self-service (użytkownik klika)
Odbiorca operacyjny (godziny, detal) · taktyczny (KPI, tygodnie) · strategiczny (trendy, kwartały)

Ten temat = narzędzia programistyczne (Jupyter, Quarto, Streamlit, LaTeX), w opozycji do ekosystemu BI (Business Intelligence — Power BI, Tableau; omawiane w Porównaniu narzędzi).

Literate programming — Jupyter / R Markdown / Quarto

Idea Knutha: kod + narracja + wynik w jednym dokumencie.

  • Jupyter — interaktywny interfejs do Pythona/R/Julia (kernels), format .ipynb (JSON — JavaScript Object Notation). Eksport nbconvert → HTML (HyperText Markup Language), PDF (Portable Document Format, przez LaTeX), slajdy.
  • R Markdown — .Rmd z chunkami R; knitr + Pandoc; wyjścia html_document, pdf_document, word_document.
  • Quarto — następca R Markdown działający z Pythonem, R, Julią, Observable; CLI (Command Line Interface) quarto render. Pliki .qmd (lepszy diff w git), wsparcie dla książek, slajdów (reveal.js), matematyki.

Pułapka Jupytera: stan kernela ≠ kolejność komórek (komórki uruchamiane poza kolejnością). Lekarstwo: „Restart & Run All" lub renderowanie z CLI (quarto render, papermill, jupyter nbconvert --execute).

Parametryzacja i automatyzacja

Raport parametryzowany — szablon z wejściami (region, miesiąc) generujący warianty.

  • Papermill — wykonuje notatnik z parametrami: papermill report.ipynb out.ipynb -p region "EU".
  • Quarto params — sekcja params: w nagłówku YAML (YAML Ain't Markup Language); dostęp zależny od języka: params$region w R, params['region'] w Pythonie.
  • Orkiestracja: cron (proste), Airflow / Prefect / Dagster (zależności, monitoring).
  • Dystrybucja: push (mail, Slack) vs pull (portal); alerty progowe (exception reporting — wyślij gdy KPI przekroczy próg).

LaTeX dla raportów naukowych

System składu Knutha; .tex → PDF. Mocne strony: typografia matematyczna, automatyczne odnośniki, bibliografia (biblatex). Klasy: article, report, beamer (slajdy). Częsta ścieżka: Quarto/R Markdown → LaTeX → PDF (autor pisze Markdown, jakość wyjścia LaTeX-owa); czysty LaTeX zostaje dla prac dyplomowych i publikacji.

Interaktywne dashboardy

  • Streamlit — model „skrypt re-runuje się od góry przy każdej interakcji"; cache @st.cache_data. Najszybszy do prototypów.
  • Plotly Dash — model callback-owy (jak React); produkcyjny, dla złożonych aplikacji.
  • Shiny (R, od 2022 też Python) — reaktywny model obliczeń, standard w R.
  • Voilà — konwertuje notatnik Jupyter w aplikację webową (widgety ipywidgets).

Wybór: prototyp eksploracyjny → Streamlit; produkcyjny dashboard → Dash; ekosystem R → Shiny.

Pixel-perfect i raporty regulacyjne

Gdy wymagany dokładny układ (faktury, sprawozdania regulacyjne — np. MiFID, Markets in Financial Instruments Directive; ESEF, European Single Electronic Format):

  • ReportLab — Python, niskopoziomowa kontrola PDF;
  • WeasyPrint — HTML + CSS (Cascading Style Sheets) → PDF;
  • JasperReports / Crystal Reports / SSRS (SQL Server Reporting Services) — silniki korporacyjne z designerem.

Format XBRL (eXtensible Business Reporting Language) — XML do sprawozdawczości, odbiorcą jest parser.

Dobre praktyki

  • Reproducibility — zamrożenie środowiska (requirements.txt / renv.lock, Docker), set_seed(), dane wersjonowane (snapshot ID lub narzędzia typu DVC — Data Version Control).
  • Single source of truth dla metryk — semantic layer (np. dbt metrics); bez niego każdy dział liczy „revenue" inaczej.
  • Bezpieczeństwo — Row-Level Security (RLS — filtr wierszy zależny od użytkownika), audyt dostępu, podpis raportów regulacyjnych.
  • Świeżość i wydajność — etykieta „dane na dzień X", pre-agregacje (materialized views), kolumnowe formaty (Parquet); nie pchaj zapytań ad-hoc na detal z dashboardu.

Powiązania z innymi zagadnieniami

  • Modele wielowymiarowe / Porównanie modelowania — kostki OLAP (Online Analytical Processing) i operacje (drill-down, roll-up, slice, pivot) to formalna podstawa raportów wielowymiarowych.
  • Porównanie narzędzi — Power BI / Tableau / Looker to gałąź low-code, alternatywna do programistycznej.
  • Wizualizacja — wykresy są budulcem każdego raportu; te same biblioteki (Matplotlib, Plotly) zasilają dashboardy.
  • Preprocessing — jakość raportu = jakość ETL (Extract, Transform, Load); czyszczenie poprzedza prezentację.
